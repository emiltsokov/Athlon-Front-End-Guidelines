Athlon Front-end Guidelines

Introduction

This document describes rules and recommendations for developing rich front-end applications, which are easy to maintain. The goal is to define guidelines, enforcing consistent style and formatting and help developers avoid common pitfalls and mistakes.

General

The three main cornerstones of our philosophy in front-end engineering are:

  1. Separation of presentation, content, and behavior.
  2. Markup should be well-formed, semantically correct and generally valid.
  3. Javascript should progressively enhance the experience.

General Practices

Indentation

For all code languages, we require indentation to be done via soft tabs (using the space character). Hitting Tab in your text editor shall be equivalent to four spaces.

Readability vs Compression

We prefer readability over file-size savings when it comes to maintaining existing files. Plenty of whitespace is encouraged. There is no need for any developer to purposefully compress HTML or CSS, nor obfuscate JavaScript.

We will use server-side or build processes to automatically minify and gzip all static client-side files, such as CSS and JavaScript, preferably when delivering the project.

HTML markup

Markup defines the structure and outline of a document and offers a structured content. Markup is not intended to define the look and feel of the content on the page beyond rudimentary concepts such as headers, paragraphs, and lists. The presentation attributes of HTML have all been deprecated and style should be contained in style sheets.

  1. Be as semantic as possible e.g. <blockquote>, <em>, <abbr>, <p>, lists, correctly ordered headers (h1 followed by h2 followed by h3), microformats.
  2. Items in list form should always be housed in a UL, OL, or DL, never a set of DIVs or Ps.
  3. Use label fields to label each form field, the for attribute should associate itself with the input field, so users can click the labels.
  4. Do not use the size attribute on your input fields. The size attribute is relative to the font-size of the text inside the input. Instead use css width property.
  5. Place an html comment on some closing div tags to indicate what element you're closing. It will help when there is lots of nesting and indentation.
  6. Tables shouldn't be used for page layout.
  7. Make use of THEAD, TBODY, and TH tags (and Scope attribute) when appropriate.
    01.<table summary="This is a chart of year-end returns for 2005.">
    02.    <thead>
    03.        <tr>
    04.            <th scope="col">Table header 1</th>
    05.            <th scope="col">Table header 2</th>
    06.        </tr>
    07.    </thead>
    08.    <tbody>
    09.        <tr>
    10.            <td>Table data 1</td>
    11.            <td>Table data 2</td>
    12.        </tr>
    13.    </tbody>
    14.</table>
  8. Always use title-case for headers and titles. Do not use all caps or all lowercase titles in markup, instead apply the CSS property text-transform:uppercase/lowercase.
  9. Use only lowercase for HTML tags and attributes.
  10. Don't leave white space between the the opening and closing tag and the tag contents.
  11. Provide alternative contents for multimedia.("alt" attributes for images, images for Adobe Flash/HTML5 Canvas animations).
  12. Avoid inline/embedded CSS code, use external stylesheets where possible.
  13. Always put your styles up top and scripts down bottom.
  14. Correct use of IDs and classes. Only use ID if there will only ever be one instance of an element on the page. IDs don't have any extra advantages over classes.
  15. Don’t put a class on an element when you can easily traverse to it in the DOM using tag names. E.g. use “.navigation UL LI”, not “.navigation. .navigation-list .navigation-item”.
  16. Don't use depricated HTML elements like <font>, <u>, <center>
  17. Don't use HTML elements' attributes like "title" or "rel" for storing information, used by scripts. Instead use custom "data-" attributes.
  18. It's strictly forbidden to nest a form inside another form.

Formatting rules

  1. Use a new line for every block, list, or table element, and indent every such child element. Inline elements should be written on the same line.

    <blockquote>
    <p><em>Space</em>, the final frontier.</p>
</blockquote>
  2. When quoting attributes values, use double quotation marks.
  3. Separate words in ID and class names by a hyphen, in order to improve understanding and scannability.
  4. Avoid iframes, as they are the most costly element to add to a given page. They block the page from firing the onload event until they are complete.

CSS

  1. Use meaningful or generic ID and class names. It is always preferable to name something, be it an ID or a class, by the nature of what it is rather than by what it looks like.
  2. Use ID and class names that are as short as possible but as long as necessary.
  3. When specifying color values, use 3 character hexadecimal notation where possible(#ffffff -> #fff).
  4. Omit unit specification after "0" values.
  5. Omit leading "0"s in values.
  6. Use shorthand properties where possible.
  7. Avoid qualifying ID and class names with type selectors, when DIV element is used. 
    #example {}
.error {}
  8. Never reference js- prefixed class names from CSS files. js- are used exclusively from JS files. Use the is- prefix for state rules that are shared between CSS and JS.
  9. Use px for font-size, because it offers absolute control over text and IE6 is largely considered deprecated. Additionally, unit-less line-height is preferred because it does not inherit a percentage value of its parent element, but instead is based on a multiplier of the font-size.
  10. Where possible, avoid expensive CSS selectors. For example, avoid the * wildcard selector
  11. All id and class declarations in CSS shall be written in only lowercase.

    12. Formatting rules

    • At minimum, format CSS with selectors on one line and each property on its own line. The declarations are indented. Also, if you specify multiple selectors, it's a good idea to start each on new line. This prevents lines from growing long and improves readability as well as version control workflow.
    • Put spaces after : in property declarations.
    • Put spaces before { in rule declarations.
    • Use a semicolon after every declaration, the last one too.
    • Avoid using @import directive, because the browser is incapable of downloading the asset in parallel, which can block the download of other assets.
    • Avoid user agent detection as well as CSS "hacks" — try a different approach first.
    • Typically a print CSS file is added to the project, which includes additional styling for printed web pages. It is a good practice to show the full links, when a web page is being printed: 
      @media print {
     a[href]:after { " (" attr(href) ") "; }
}
    • Group sections by a section comment, when needed.
    • We use the following as guideline for our declaration order: 
      element.class {
	/* Positioning */
	display: inherit;
	float: none;
	clear: none;
	visibility: visible;
	overflow: visible;
	position: static;
	top: 0;
	right: 0;
	bottom: 0;
	left: 0;
	z-index: 1;
	
	/* Margin, padding and box model */
	width: auto;
	min-width: auto;
	max-width: auto;
	height: auto;
	min-height: auto;
	max-height: auto;
	margin: 0;
	padding: 0;
	border: none;
	
	/* Background */
	background: none;
	
	/* Text and font properties */
	color: #000;
	direction: inherit;
	letter-spacing: inherit;
	line-height: inherit;
	text-align: inherit;
	text-transform: inherit;
	font-size: inherit;
	font-weight: inherit;
	/*...*/
	
	/*Element specific properties - Lists, Tables */
	list-style: inherit;
	border-collapse: inherit;
	border-spacing: inherit;
	cursor: auto;
	opacity: 1;
}

JS

JavaScript code, when properly applied to a web page, enhances the overall user and browser-based experience through attaching to events and controlling the overall behavior layer.

We primarily develop new applications in jQuery, though we have expertise in plain JavaScript as well as all major modern javascript libraries.

General principles

  1. 99% of code should be housed in external javascript files. They should be included at the END of the BODY tag for maximum page performance.
  2. Don't rely on the user-agent string. Do proper feature detection.
  3. Don't use document.write(), avoid eval(), void, "with" and "continue" statements.
  4. All Boolean variables should start with "is".
  5. Name variables and functions logically: For example: popUpWindowForAd rather than myWindow.
  6. Don't manually minify. With the exception of the traditional i, etc. for for loops, variables should be long enough to be meaningful.
  7. Constants or configuration variables (like animation durations, etc.) should be at the top of the file.
  8. Strive to create functions which can be generalized, take parameters, and return values. This allows for substantial code reuse and, when combined with includes or external scripts, can reduce the overhead when scripts need to change. For example, instead of hard coding a pop-window with window size, options, and url, consider creating a function which takes size, url, and options as variables.
  9. Comment your code! It helps reduce time spent troubleshooting JavaScript functions.
  10. Minimize global variables - the less globals you create, the better.
  11. All JavaScript variables shall be written in either completely lowercase letter or camelCase. The one exception to this are Constructor functions, which are capitalized by tradition.
  12. Avoid unnecessary DOM manipulations

Formatting rules

  1. Use soft-tabs with a four space indent.
  2. Use implicit parentheses when possible. 
    if (something) {
	//Use implicit parentheses, though not required by JS syntax
}
  3. In general, the use of whitespace should follow longstanding English reading conventions. Such that, there will be one space after each comma and colon (and semi-colon where applicable), but no spaces immediately inside the right and left sides of parenthesis. In short, we advocate readability within reason. Additionally, braces should always appear on the same line as their preceding argument. 
    for (var i = 0, j = arr.length; i < j; i++) {
	//Do something.
}
  4. In jQuery use $.fn.on instead of $.fn.bind, $.fn.delegate and $.fn.live.
  5. When possible try to prefix all javascript-based selectors with js-. This is taken from slightly obtrusive javascript. The idea is that you should be able to tell a presentational class from a functional class.

Media and other assets

Images

  1. Use CSS backgrounds for decorative images and <img> tags for content images, when possible.
  2. For repeating images, use something larger than 1x1 pixels
  3. You should never be using spacer images, except when designing HTML newsletters.
  4. Use CSS sprites generously. They make hover states easy, improve page load time.
  5. When exporting an image for web, make sure you choose the appropriate format, so that the result is visually appealing and optimised in file size at the same time (JPG, PNG, GIF).
  6. Typically, all images should be sliced with a transparent background (PNG8). All should be cropped tightly to the image boundaries. However, the logo should always have a background matte and have padding before the crop. (so other people can hotlink to the file)
  7. Typically, before delivering the project an automated optimization is done via an image optimization software.
  8. Before delevery, don't forget to add favicon to the project. It can be easily created online.

Video

  1. When exporting video content for native HTML5 playback, bear in mind that there are two formats, which cover the majority of browsers out there:
    • MP4 H.264 (with AAC or MP3 audio)
    • WebM (with OGG audio)

    * Bear in mind that the audio and video HTML5 elements are not supported in IE8-.

  2. Always provide alternative content (like images/text/Adobe Flash) as fallback option.

Fonts

The @font-face at-rule allows you to define custom fonts. It was first defined in the CSS2 specification, but was removed from CSS2.1. Currently, it's a draft recommendation for CSS3.

Our first and most preferred choice for customizing fonts on the web is @font-face, simply because it is part of the CSS Fonts Module working draft which means it will continue to grow in popularity as browser support grows, and ease of use for it improves as it becomes more stable.

@font-face {
	font-family: 'MyFontFamily';
	src: url('myfont-webfont.eot');                      /* IE9 Compat Modes */
	src: url('myfont-webfont.eot?iefix') format('eot'),  /* IE6-IE8 */
	url('myfont-webfont.woff') format('woff'),           /* Modern Browsers */
	url('myfont-webfont.ttf') format('truetype'),        /* Safari, Android, iOS */
	url('myfont-webfont.svg#svgFontName') format('svg'); /* Legacy iOS */
	font-weight: normal;
	font-style: normal;
}

Some tips for using @font-face include:

  1. IE 6–8 will only accept a TrueType font packaged as an EOT.
  2. font-weight and font-style have different meanings within @font-face. Declarations where font-weight:bold; means this is the bold version of this typeface, rather than apply bold to this text
  3. We use automated tools for creating our different font formats, like FontSquirrel or use other services such as Google WebFonts API or Typekit, when implementing custom font into our projects.

    4. Font Licensing

    Even though you can transform just about any font into a web font file, you should still make sure it is legally okay for you to do so. Many foundries have updated their conditions to specify how their fonts can be used on the web.
